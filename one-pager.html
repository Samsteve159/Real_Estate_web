Manifest RE — AI Companion Site

Local-first · deploys to Cloudflare
Status: built & live-tested · Phase 4 (deploy)
Looking for: data-source & modelling feedback

Summary

A standalone web app that augments an independent real-estate agent's site with two AI features: an Instant Home Valuation lead magnet and an AI Buyer Concierge chatbot. Built local-first (no dependency on the agent's locked WordPress site), deploys free to Cloudflare. All Claude API calls are server-side. The open questions for you are about data sourcing and valuation methodology — flagged in red at the bottom.

Data assets

  • listings.json — current properties seeded from the agent's public pages (address, suburb, price/guide, beds/baths/cars, type, status, sold price where shown).
  • suburbs.json — serviced suburbs + median/context notes (sourcing TBD — see Q1).
  • leads.db — captured leads: name, contact, source tool, intent, payload (estimate or chat transcript), timestamp.

The two tools

🏡 Instant Home Valuation

Visitor submits address + attributes (beds/baths/cars, type).
API selects relevant recently-sold comps from listings.json + suburb context.
Claude reasons over comps → returns structured JSON: low, high, midpoint, confidence, rationale, comparables_used.
UI shows an "indicative estimate"; CTA → real appraisal = lead capture to D1.

Note: this is a lead-gen heuristic, not a calibrated AVM. Honesty in copy is deliberate.

💬 AI Buyer Concierge

Chat widget streams to /api/chat.
System prompt scoped to Manifest RE + current listings injected as context (light RAG, no vector DB initially).
Claude tool use: search_listings(filters) (NL search) + capture_lead().
Books inspection / appraisal → lead to D1.

Scale path: if listing volume outgrows context window, move to embeddings + vector search (Cloudflare Vectorize).

Open questions — your feedback wanted

Q1 Grounding the valuation

Comps from the agent's own sold listings are thin. Is there a sensible free/cheap source of suburb-level sold data or medians (e.g. state valuer-general open data, ABS, scraping ethics)? Or do we keep it strictly comp-based and lean harder on the "indicative" framing?

Q2 Valuation methodology

Is "LLM reasons over a handful of comps" defensible as a lead-gen estimate, or should we put a simple regression / nearest-neighbour baseline underneath and use the LLM only for the rationale text? Where's the line on accuracy claims before it's misleading?

Q3 Listing data freshness

Seeding via one-off scrape of public pages gets stale. Worth building a scheduled refresh (Cloudflare Cron), and is scraping the right approach vs. asking the agent/provider for a feed (XML/portal export, e.g. REAXML)?

Q4 Lead data & privacy

We're storing PII (name, contact, address) + chat transcripts. Partly addressed in the security review (input caps + validation, CORS allowlist, stats gating, server-side key). Still open before go-live: retention policy, consent copy, and Australian Privacy Principles compliance.
